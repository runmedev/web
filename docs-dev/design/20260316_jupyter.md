# 20260316: Jupyter Server Integration via Runme Proxy

## Context

Related issue: https://github.com/runmedev/web/issues/63

Issue #63 describes two backend challenges for ipykernel support:

1. Kernel process management.
2. WebSocket <-> ZeroMQ bridging.

For this design pass, we choose a simpler integration point: reuse `jupyter server`
instead of implementing ZeroMQ bridging directly in Runme.

Assumption for v0: user starts Jupyter Server manually:

```bash
jupyter server --no-browser
```

## Decision

Use Runme as a proxy/control plane in front of an existing Jupyter Server.

- Browser talks only to Runme.
- Runme talks to Jupyter HTTP + WebSocket APIs.
- Jupyter continues to own kernel/channel internals.
- Frontend translates Jupyter protocol messages into Runme notebook `CellOutput` updates.
- Runme server remains thin: auth/token storage, kernel lifecycle APIs, and WebSocket proxying.
- Runme does not use Jupyter Sessions API in v0; frontend owns cell/kernel selection state.

This avoids implementing ZeroMQ in Go in the first milestone and gives us a fast
path to a usable Python REPL experience.

Server lifecycle split for v0:

- Start/stop/list Jupyter server with Jupyter CLI (`jupyter server ...`).
- Start/stop/list kernels with Jupyter Server REST API (`/api/kernels`) via Runme backend.

## Design Principle: Thin Jupyter Proxy

For Jupyter integration, Runme should be a thin proxy layer, not a second
Jupyter implementation.

- Runme is responsible for:
  - authn/authz,
  - server selection/routing,
  - token/header injection,
  - allowlisted endpoint exposure,
  - transport proxying (HTTP + WebSocket).
- Runme is not responsible for:
  - interpreting or rewriting Jupyter protocol payloads,
  - translating kernel channel message semantics,
  - re-implementing Jupyter execution state machines.

Practical rule:

- be route-aware and policy-aware,
- remain payload-agnostic for proxied Jupyter requests/responses whenever possible.

## Goals (v0)

- Execute Python cells in Runme notebooks via a user-started local Jupyter Server.
- Keep browser transport uniform (Runme endpoint remains the single backend entry).
- Support the CUJ flow in `docs-dev/CUJs/jupyter.md`, including shared REPL state across cells.
- Keep credentials/tokens out of notebook file content.

## Non-goals (v0)

- Runme-managed kernel lifecycle (auto-start/stop/restart).
- Full parity with all Jupyter features (widgets/comm/custom extensions).
- Multi-user/multi-tenant hardening.
- Replacing Runme execution path for non-Jupyter runners.

## Minimal CUJ

The acceptance scenario is defined in:

- `docs-dev/CUJs/jupyter.md`

## Proposed Architecture (v0)

### Components

1. Web app:
   - Adds/selects a `Jupyter Server` runner target.
   - Creates/selects `Kernel runners` backed by that server.
   - Sends `execute_request` on Jupyter channels via Runme WebSocket proxy.
   - Translates Jupyter channel messages into notebook `CellOutput` state.
2. Runme backend:
   - Loads Jupyter server config files from `${RUNME_CONFIG_DIR}/jupyter/`.
   - Caches server configs in memory (reload on cache miss).
   - Creates/reuses kernels via Jupyter kernel APIs.
   - Proxies Jupyter kernel channels WebSocket frames.
   - Exposes kernel lifecycle management routes.
3. Jupyter Server:
   - Handles kernel lifecycle and message routing to ipykernel.

### Runner Model (v0)

- `Jupyter Server record`: file-backed connection config (base URL + token + runner).
- `Kernel runner`: executable target bound to one running kernel id from that server.

Proposed App Console surface:

- `jupyter.servers.get()`
- `jupyter.kernels.start(serverName, options)`
- `jupyter.kernels.get(serverName)`
- `jupyter.kernels.stop(serverName, kernelNameOrId)`

Expected user actions:

1. Sync server config files on the Runme host (`${RUNME_CONFIG_DIR}/jupyter/port-${PORTNUM}.json`).
2. Start kernel via Runme backend client.
3. Use kernel runner in notebook cells.
4. Stop kernel when done.

### Runme Implementation Boundary

- `AppKernel` remains the browser-local JavaScript execution path.
- Jupyter integration should be implemented in backend Runme code as a small
  Jupyter client/proxy library (HTTP + kernel channels WebSocket).
- That backend library owns calls for:
  - create/list/delete kernels,
  - server+kernel lookup/cache state in Runme,
  - proxying channels traffic without translating message payloads.

Frontend owns:

- building/sending `execute_request`,
- parsing Jupyter channel replies (`stream`, `execute_result`, `display_data`, `error`, etc.),
- mapping those replies into Runme notebook output state.

## Jupyter Object Model (Server, Session, Kernel)

- `Server`: one Jupyter Server process that exposes REST + WebSocket APIs.
- `Session`: mapping between a client document context (for example notebook path/name) and a kernel.
- `Kernel`: the actual Python REPL process that executes code.

Implication:

- A single Jupyter Server can handle multiple sessions.
- Sessions can point to distinct kernels (independent REPLs), which is the
  normal behavior when users open multiple notebooks.
- Multiple clients can also attach to the same kernel/session if explicitly configured.

For Runme v0, the target behavior is:

- do not create/use Jupyter sessions,
- kernels are reusable across notebooks/tabs (not document-bound).
- users may select the same kernel in multiple notebooks.
- execution controls for Jupyter cells expose:
  - `Language` (`ipython`)
  - `Runner` (host/proxy runner)
  - `Kernel` (kernel runner on selected server)

### Data flow (single execution)

1. User writes/updates server config JSON on the Runme host:
   `${RUNME_CONFIG_DIR}/jupyter/port-${PORTNUM}.json`.
2. Runme backend calls Jupyter kernel APIs to ensure an active kernel.
3. Runme backend opens Jupyter kernel channels WebSocket.
4. Web app opens dedicated Runme Jupyter channels route:
   `/v1/jupyter/servers/{server}/kernels/{kernel_id}/channels`.
5. Web app sends native Jupyter channels messages (including Jupyter
   `execute_request`) over that WebSocket.
6. Jupyter server streams IOPub/shell replies over channels.
7. Runme forwards frames to web app unchanged (protocol passthrough).
8. Web app maps Jupyter channel replies into `CellOutput` updates
   (including streaming stdout/stderr and rich outputs).

## Proxy Behavior (v0)

- Runme is intentionally protocol-thin for channels:
  - terminates/authenticates client connection,
  - attaches Jupyter auth upstream,
  - forwards channel frames bidirectionally,
  - does not transform Jupyter message payloads.
- Runme does not expose ZeroMQ or kernel connection files to the browser.
- Jupyter channels WebSocket route is protected by IAM auth and requires
  caller authorization for the `runner` role.

## Backend API Contract (v0)

Decision: use a constrained proxy contract with minimal protocol awareness.

- Runme exposes explicit allowlisted routes for Jupyter kernel lifecycle operations.
- Runme does not expose a generic unrestricted `/jupyter-proxy` endpoint.
- For allowed operations, Runme forwards requests/responses to Jupyter with
  minimal transformation.

### Routes

Server config:

- `GET /v1/jupyter/servers`

Kernel lifecycle:

- `POST /v1/jupyter/servers/{server}/kernels`
  - upstream: `POST /api/kernels`
- `GET /v1/jupyter/servers/{server}/kernels`
  - upstream: `GET /api/kernels`
- `DELETE /v1/jupyter/servers/{server}/kernels/{kernel_id}`
  - upstream: `DELETE /api/kernels/{kernel_id}`
- `POST /v1/jupyter/servers/{server}/kernels/{kernel_id}:interrupt`
  - upstream: `POST /api/kernels/{kernel_id}/interrupt`
- `POST /v1/jupyter/servers/{server}/kernels/{kernel_id}:restart`
  - upstream: `POST /api/kernels/{kernel_id}/restart`

Channels WebSocket:

- `GET /v1/jupyter/servers/{server}/kernels/{kernel_id}/channels` (WebSocket upgrade)
  - upstream: `GET /api/kernels/{kernel_id}/channels` (WebSocket)

### Route Wrapper Definitions

All wrappers follow this pattern:

1. Resolve `{server}` to stored server config (`base_url`, auth material, runner binding).
2. Enforce IAM auth + `runner` role.
3. Build upstream Jupyter URL from resolved `base_url` + mapped path.
4. Inject upstream auth (token/header) server-side.
5. Proxy request/response with minimal transformation.

`GET /v1/jupyter/servers`

- Purpose: list configured server records.
- Source of truth: `${RUNME_CONFIG_DIR}/jupyter/*.json`.
- Response: list of server records with secrets redacted (no raw token values).
- v0 naming convention: `{server} = port-${PORTNUM}`.
- v0 server config path:
  - `${RUNME_CONFIG_DIR}/jupyter/${SERVER}.json`

`POST /v1/jupyter/servers/{server}/kernels`

- Upstream mapping: `POST /api/kernels`.
- Request body: pass-through Jupyter kernel-create payload.
  - Default when omitted: `{}`.
  - Common payload:

```json
{
  "name": "python3",
  "path": ""
}
```

- Response: pass-through Jupyter kernel model (`id`, `name`, `last_activity`, `execution_state`, `connections`).

`GET /v1/jupyter/servers/{server}/kernels`

- Upstream mapping: `GET /api/kernels`.
- Response: pass-through list of Jupyter kernel models.

`DELETE /v1/jupyter/servers/{server}/kernels/{kernel_id}`

- Upstream mapping: `DELETE /api/kernels/{kernel_id}`.
- Response: pass-through status (`204` on success).

`POST /v1/jupyter/servers/{server}/kernels/{kernel_id}:interrupt`

- Upstream mapping: `POST /api/kernels/{kernel_id}/interrupt`.
- Response: pass-through status (`204` on success).

`POST /v1/jupyter/servers/{server}/kernels/{kernel_id}:restart`

- Upstream mapping: `POST /api/kernels/{kernel_id}/restart`.
- Response: pass-through Jupyter kernel model.

`GET /v1/jupyter/servers/{server}/kernels/{kernel_id}/channels` (WebSocket)

- Upstream mapping: `GET /api/kernels/{kernel_id}/channels` (WebSocket).
- Behavior:
  - bidirectional frame forwarding,
  - no protocol payload transformation,
  - close/error propagation in both directions.

Identifier rule for HTTP routes:

- kernel lifecycle routes are keyed by `kernel_id` (Jupyter canonical id).
- Jupyter channels WebSocket route is also keyed by `kernel_id`.
- Frontend maintains a kernel name -> `kernel_id` mapping for display and user selection.
- Frontend/App Console may accept friendly kernel names, but must resolve
  name -> `kernel_id` before invoking any kernel lifecycle HTTP route or channels WS route.

### Request/Response Rules

- HTTP JSON request/response bodies for kernel operations are passed through
  with minimal transformation (prefer opaque forwarding).
- Runme injects upstream auth/token material from stored server config before
  forwarding any Jupyter request.
- Token is never returned by these routes.
- Upstream error status codes are preserved where possible; response details are
  sanitized for secrets.
- All Jupyter routes (HTTP + WebSocket) are protected by IAM and require
  `runner` role authorization.

### Proxy Guardrails

- `{server}` must resolve to a previously configured Jupyter server record.
- Frontend cannot supply arbitrary upstream URL/path at call time.
- Only the above allowlisted methods/routes are proxied.
- Server config requires a token value in v0; requests for servers without token
  config fail fast.

### Config Cache Behavior (v0)

- Load all `${RUNME_CONFIG_DIR}/jupyter/*.json` records at server startup.
- Resolve `{server}` from in-memory cache on request path dispatch.
- On cache miss, reload the servers directory once and retry lookup.
- No mtime/TTL polling in v0.

### Failure/Reconnect Policy (v0)

- Kernel lifetime is independent of a single frontend WebSocket connection.
- On channels WebSocket disconnect, frontend attempts reconnect to the same
  `{server, kernel_id}`.
- If reconnect succeeds, subsequent executions continue on the same kernel.
- In-flight execution at disconnect time is marked `unknown/incomplete`; v0 does
  not attempt message replay or output reconstruction.
- UI should surface a recovery hint as a cell inline banner:
  - `Connection lost; output may be incomplete. Re-run the cell.`

### Frontend Translation Behavior (v0)

- On new cell execution start, frontend clears existing outputs for that cell immediately.
- New outputs for the execution are then appended from Jupyter channel messages.
- v0 supported output categories:
  - text (`stream`, `error`)
  - images (`image/png`, `image/jpeg`, `image/svg+xml`)
  - HTML (`text/html`)
- `update_display_data` is not fully implemented with `display_id` patch semantics in v0.
  - v0 degrade behavior: append as a new rich output item.
  - this preserves visibility that updates are happening, without notebook-wide patch logic.

Message mapping table:

| Jupyter message | v0 frontend handling | Result in notebook outputs |
| --- | --- | --- |
| `stream` | Append chunk immediately to stdout/stderr buffer for active cell execution. | Streaming text output. |
| `error` | Append error payload (`ename`, `evalue`, traceback) to stderr output and mark execution failed. | Error text in output area. |
| `display_data` | Append a new rich output (mime bundle). | Rich output (HTML/images). |
| `execute_result` | Append a new rich output (mime bundle). | Rich output (HTML/images). |
| `update_display_data` | Do not patch by `display_id`; append as a new rich output item. | Additional rich output entry (degraded behavior). |
| `clear_output` | Ignore message in v0. Output clearing is handled at execution start only. | No-op for in-band clear message. |
| `input_request` | Not supported in v0. Surface unsupported-input notice and fail execution. | Cell marked failed with actionable message. |
| `status` (`busy`/`idle`) | Update cell execution lifecycle state for active execution. | Running/completed state updates. |
| shell `execute_reply` | Capture final execution status (`ok`/`error`/`abort`) for active execution. | Execution summary success/failure. |

Additional v0 rules:

- Allow multiple cells to execute concurrently on the same kernel.
- For each executing cell, process/display only messages associated with that
  cell's active execution `msg_id` (`parent_header.msg_id` match).
- `clear_output` message semantics (`wait=true/false`) are not implemented in v0.
- `input_request` stdin prompts are deferred; v0 surfaces explicit unsupported-input
  error and asks user to rerun with non-interactive code.
- Unsupported message families (`comm_*`, widget traffic, and other non-core messages) are ignored in v0.
- Completion gate:
  - mark execution complete only after both matching shell `execute_reply` and
    matching IOPub `status: idle` are observed.
  - if one signal is missing past timeout, mark cell `unknown/incomplete` and
    surface rerun guidance.

Output size limits (v0):

- max combined stdout+stderr text per cell execution: `16 MiB`
- max single rich output item payload (HTML/image mime item): `10 MiB`
- max total output payload per cell execution (all output items): `32 MiB`

Limit behavior:

- when a limit is hit, truncate additional output for that scope
- append a visible truncation notice in cell output metadata/text
- keep execution lifecycle signals (`status`, `execute_reply`) processing even after truncation

## Alternatives

### Alternative: Generic `/jupyter-proxy` Route

Expose a generic proxy endpoint that allows frontend-controlled upstream path
forwarding.

Pros:

- minimal backend route surface to implement
- maximum flexibility for frontend experimentation

Cons:

- weaker authorization granularity (harder to enforce least privilege)
- higher SSRF/path-abuse risk
- frontend couples directly to arbitrary upstream routes
- weaker operation-level observability and policy controls

Decision for v0:

- not selected; use allowlisted routes only.

### Alternative: Reuse Existing Runner WebSocket Route

Carry Jupyter channels traffic on the existing runner WebSocket route.

Pros:

- fewer new endpoints
- possible reuse of existing transport plumbing

Cons:

- existing route semantics may assume Runme runner protocol, not Jupyter channels
- increases coupling between unrelated protocols
- harder to reason about compatibility and debugging

Decision for v0:

- not selected; use dedicated Jupyter channels route:
  - `/v1/jupyter/servers/{server}/kernels/{kernel_id}/channels`

### Alternative: Backend Translates Jupyter Protocol to Runme-native Events

Backend would parse IOPub/shell messages and emit normalized Runme execution events.

Pros:

- thinner frontend protocol logic
- reusable contract for non-web clients

Cons:

- diverges from standard Jupyter frontend model
- higher backend complexity and maintenance risk
- harder to maintain parity for advanced Jupyter message types

Decision for v0: do translation in frontend and keep Runme server thin.

## State Model (v0)

Per configured Jupyter runner, Runme keeps persisted config plus in-memory runtime state.

Persisted config (`${RUNME_CONFIG_DIR}/jupyter/${SERVER}.json`):

- `base_url`
- `token`
- `runner`

In-memory runtime state:

- server kernel inventory/cache (`kernel_id`, kernel name, state)
- active upstream channel connection metadata

Config persists across restarts; runtime kernel/channel state is ephemeral and
reconstructed as clients reconnect.

## Security Notes

- Jupyter token is entered by user and retained only on the Runme backend.
- Token is not written into notebook metadata/content.
- Browser never directly opens Jupyter kernel channels.
- Persisted server config files should be written with restrictive permissions
  (owner read/write).
