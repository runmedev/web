# 20260316: Jupyter Server Integration via Runme Proxy

## Context

Related issue: https://github.com/runmedev/web/issues/63

Issue #63 describes two backend challenges for ipykernel support:

1. Kernel process management.
2. WebSocket <-> ZeroMQ bridging.

For this design pass, we choose a simpler integration point: reuse `jupyter server`
instead of implementing ZeroMQ bridging directly in Runme.

Assumption for v0: user starts Jupyter Server manually:

```bash
jupyter server --no-browser
```

## Decision

Use Runme as a proxy/control plane in front of an existing Jupyter Server.

- Browser talks only to Runme.
- Runme talks to Jupyter HTTP + WebSocket APIs.
- Jupyter continues to own kernel/channel internals.

This avoids implementing ZeroMQ in Go in the first milestone and gives us a fast
path to a usable Python REPL experience.

Server lifecycle split for v0:

- Start/stop/list Jupyter server with Jupyter CLI (`jupyter server ...`).
- Start/stop/list kernels with Jupyter Server REST API (`/api/kernels`) via Runme backend.

## Goals (v0)

- Execute Python cells in Runme notebooks via a user-started local Jupyter Server.
- Keep browser transport uniform (Runme endpoint remains the single backend entry).
- Support a minimal CUJ with a single cell (`print("hello from jupyter")`).
- Keep credentials/tokens out of notebook file content.

## Non-goals (v0)

- Runme-managed kernel lifecycle (auto-start/stop/restart).
- Full parity with all Jupyter features (widgets/comm/custom extensions).
- Multi-user/multi-tenant hardening.
- Replacing Runme execution path for non-Jupyter runners.

## Minimal CUJ

The acceptance scenario is defined in:

- `docs-dev/cujs/jupyter.md`

## Proposed Architecture (v0)

### Components

1. Web app:
   - Adds/selects a `Jupyter Server` runner target.
   - Creates/selects `Kernel runners` backed by that server.
   - Sends execute requests through existing Runme transport.
2. Runme backend:
   - Stores Jupyter connection settings for session scope (URL + token).
   - Creates/reuses a Jupyter session/kernel.
   - Opens Jupyter kernel channels WebSocket.
   - Proxies execution requests and output events.
3. Jupyter Server:
   - Handles kernel/session lifecycle and message routing to ipykernel.

### Runner Model (v0)

- `Jupyter Server runner`: connection config (base URL + token).
- `Kernel runner`: executable target bound to one running kernel id from that server.

Proposed App Console surface:

- `jupyter.servers.update(name, info)`
- `jupyter.servers.get()`
- `jupyter.kernels.start(serverName, options)`
- `jupyter.kernels.get(serverName)`
- `jupyter.kernels.stop(serverName, kernelNameOrId)`

Expected user actions:

1. Add/connect server runner.
2. Start kernel via Runme backend client.
3. Use kernel runner in notebook cells.
4. Stop kernel when done.

### Runme Implementation Boundary

- `AppKernel` remains the browser-local JavaScript execution path.
- Jupyter integration should be implemented in backend Runme code as a small
  Jupyter client/proxy library (HTTP + kernel channels WebSocket).
- That backend library owns calls for:
  - create/list/delete kernels,
  - optional create/list/delete sessions,
  - execute request/reply forwarding.

## Jupyter Object Model (Server vs Session vs Kernel)

- `Server`: one Jupyter Server process that exposes REST + WebSocket APIs.
- `Session`: mapping between a client document context (for example notebook path/name) and a kernel.
- `Kernel`: the actual Python REPL process that executes code.

Implication:

- A single Jupyter Server can handle multiple sessions.
- Sessions can point to distinct kernels (independent REPLs), which is the
  normal behavior when users open multiple notebooks.
- Multiple clients can also attach to the same kernel/session if explicitly configured.

For Runme v0, the target behavior is:

- one Jupyter session+kernel per opened notebook tab by default,
- which gives independent REPL state across two open docs.
- execution controls for Jupyter cells expose:
  - `Language` (`ipython`)
  - `Runner` (host/proxy runner)
  - `Kernel` (kernel runner on selected server)

### Data flow (single execution)

1. User configures Jupyter server URL/token in Runme UI.
2. Runme backend calls Jupyter session APIs to ensure an active kernel.
3. Runme backend opens Jupyter kernel channels WebSocket.
4. Web app requests cell execution via Runme.
5. Runme forwards execute request on Jupyter channels.
6. Runme receives IOPub/shell replies and forwards normalized outputs to web app.
7. Web app updates notebook cell output state.

## Proxy Behavior (v0)

- Runme is protocol-aware enough to:
  - submit execute requests,
  - forward stream/error/result payloads,
  - map completion/failure to existing notebook execution states.
- Runme does not expose ZeroMQ or kernel connection files to the browser.

## State Model (v0)

Per configured Jupyter runner, Runme keeps in-memory session state:

- `base_url`
- `token` (or auth header material)
- `session_id`
- `kernel_id`
- `channels_ws_url`

State is ephemeral for v0 (lost on backend restart).

## Security Notes

- Jupyter token is entered by user and retained only on the Runme backend session.
- Token is not written into notebook metadata/content.
- Browser never directly opens Jupyter kernel channels.

## Rollout Plan

1. Land minimal CUJ and design docs (this change).
2. Implement local Jupyter runner config UI + backend proxy wiring.
3. Pass CUJ end-to-end with one Python cell.
4. Extend coverage to richer outputs and kernel/session lifecycle controls.

## Open Questions

- Runner config UX/API naming:
  - dedicated `Jupyter` runner type,
  - or extension of existing runner config model?
- Kernel reuse policy beyond v0:
  - keep strict one-kernel-per-notebook semantics,
  - allow opt-in shared kernel/session,
  - or expose explicit attach/detach controls?
- Failure policy:
  - reconnect behavior when Jupyter server restarts?
